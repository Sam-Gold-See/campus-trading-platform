# 动态积分引擎与信用流水 - 后端技术设计文档

## 1. 概述

本文档详细描述校园交易平台动态积分引擎与信用流水的后端技术实现方案，包括积分规则引擎、信用分自动重算、信用流水记录等功能模块的技术架构、接口设计、数据模型和安全策略。

**核心设计原则**：所有信用分变动必须可追溯。通过预定义积分规则表驱动分数计算，新增行为类型只需新增规则配置，无需改动代码。每次评价提交后异步触发信用分重算。

## 2. 技术栈

- **框架**: Spring Boot 4.x
- **安全**: Spring Security + JWT
- **数据库**: MySQL 8.0+
- **缓存**: Redis
- **异步**: Spring Async（`@Async`）

## 3. 核心功能模块设计

### 3.1 积分规则引擎

**业务目标**：根据用户交易行为自动增减信用分，规则可配置、可追溯。

#### 3.1.1 积分规则定义

| 行为 | 评分类型 | 分值变化 | 触发时机 |
|:-----|:---------|:---------|:---------|
| 完成一次交易（成交） | 成交奖励 | **+1** | 标记成交时（双方各自 +1） |
| 收到好评 | 评价加分 | **+2** | 评价提交时（好评触发） |
| 收到中评 | 评价影响 | **0** | 评价提交时（不增不减） |
| 收到差评 | 评价扣分 | **-2** | 评价提交时（差评触发） |
| 被举报"放鸽子"且核实 | 违规扣分 | **-5** | 管理员审核通过后触发 |
| 新用户注册 | 初始赋值 | **100** | 注册完成时 |

#### 3.1.2 规则引擎设计

##### 规则配置表（可选扩展，初期代码硬编码）
```sql
CREATE TABLE credit_rule (
    id INT PRIMARY KEY AUTO_INCREMENT,
    action_code VARCHAR(50) NOT NULL COMMENT '行为编码',
    action_name VARCHAR(100) NOT NULL COMMENT '行为名称',
    score_change INT NOT NULL COMMENT '分值变化（正数加分，负数扣分）',
    is_active TINYINT DEFAULT 1 COMMENT '是否启用',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

**初期实现**：规则较少（6 条），直接在 Java 代码中用枚举定义，简单直观。后续规则增多时迁移到数据库配置表。

##### 积分规则枚举设计
```java
public enum CreditActionEnum {
    DEAL_COMPLETE("deal_complete", "成交奖励", +1),
    GOOD_REVIEW("good_review", "收到好评", +2),
    NEUTRAL_REVIEW("neutral_review", "收到中评", 0),
    BAD_REVIEW("bad_review", "收到差评", -2),
    NO_SHOW_VERIFIED("no_show_verified", "放鸽子核实", -5),
    INIT("init", "新用户初始分", +100);

    private final String actionCode;
    private final String actionName;
    private final int scoreChange;
}
```

### 3.2 信用流水分表

**业务目标**：记录每一次信用分变动的来源和原因，实现分数的完全可追溯。

#### 3.2.1 信用流水表（新建）

```sql
CREATE TABLE credit_log (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL COMMENT '用户ID',
    action_code VARCHAR(50) NOT NULL COMMENT '行为编码',
    score_change INT NOT NULL COMMENT '分数变化',
    score_before INT NOT NULL COMMENT '变动前分数',
    score_after INT NOT NULL COMMENT '变动后分数',
    ref_id BIGINT NULL COMMENT '关联ID（评价ID/帖文ID）',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,

    INDEX idx_user_id (user_id),
    INDEX idx_created_at (created_at)
);
```

#### 3.2.2 流水记录示例

| id | user_id | action_code | score_change | score_before | score_after | ref_id |
|:---|:--------|:------------|:-------------|:-------------|:------------|:-------|
| 1 | 5 | init | +100 | 0 | 100 | NULL |
| 2 | 5 | deal_complete | +1 | 100 | 101 | 23 |
| 3 | 5 | good_review | +2 | 101 | 103 | 7 |
| 4 | 5 | bad_review | -2 | 103 | 101 | 9 |

### 3.3 信用分重算流程

**业务目标**：评价提交后异步重算被评价人的信用分。

#### 3.3.1 重算流程
```
评价提交成功
  → 异步触发 CreditScoreRecalculator
    → 1. 查询被评价人当前信用分
    → 2. 根据 rating_type 计算分数变化
        - 好评(+2) / 中评(0) / 差评(-2)
    → 3. 计算新分数：newScore = currentScore + scoreChange
    → 4. 更新 user.credit_score = newScore
    → 5. 写入 credit_log 流水记录
    → 6. 检查是否需要变更 user_status
        - newScore < 60 → user_status = 1（禁用）
        - newScore >= 60 且之前被禁用 → user_status = 0（恢复）
    → 7. 更新 Redis 缓存信用分
```

#### 3.3.2 同步更新 vs 异步重算

| 方式 | 优点 | 缺点 |
|:-----|:-----|:-----|
| 同步（评价提交时直接更新） | 实时，实现简单 | 阻塞评价提交响应 |
| 异步（@Async 重算） | 不阻塞评价提交，解耦 | 短暂的数据不一致 |
| **推荐：同步更新 + 异步写流水** | 信用分实时生效 | 流水写入延迟可接受 |

**初期方案**：同步更新 `user.credit_score`，异步写入 `credit_log`。分数更新是单字段原子操作，耗时极短（< 1ms），不影响用户体验。

#### 3.3.3 信用分底限保护
```java
// 信用分最低保护（不低于 0）
int newScore = Math.max(0, currentScore + scoreChange);

// 信用分无上限（优秀用户可超过 100）
```

### 3.4 信用流水查询模块 (`/api/user/credit-logs`)

#### 3.4.1 API 设计

##### 查询信用流水
- **接口**: `GET /api/user/credit-logs`
- **请求头**: `Authorization: Bearer {token}`
- **请求参数**:
```
page: integer  // 页码（默认1）
```

- **响应参数**:
```json
{
  "code": 200,
  "data": {
    "list": [
      {
        "id": "long",
        "actionName": "string",    // "收到好评" / "成交奖励" / "收到差评"

  }
}
```

## 5. 安全设计

### 5.1 积分安全
- 信用分更新使用原子操作（`UPDATE user SET credit_score = ? WHERE id = ?`）
- 流水记录不可删除（防止篡改记录）
- 更新前查询当前分数，防止并发覆盖（乐观锁可后续引入）

### 5.2 权限控制
- 信用流水仅用户本人和管理员可查看
- 新用户注册时 credit_score 写死为 100（不从前端接收）
