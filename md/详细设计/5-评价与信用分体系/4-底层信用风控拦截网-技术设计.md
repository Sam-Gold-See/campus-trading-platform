# 底层信用风控拦截网 - 后端技术设计文档

## 1. 概述

本文档详细描述校园交易平台底层信用风控拦截网的后端技术实现方案，通过在业务接口层建立统一的风控拦截机制，对信用分低于阈值的用户强制权限剥夺，保障大盘交易环境安全。

**核心设计原则**：被动防御、最小侵入。风控逻辑集中在 AOP 切面或拦截器中，业务代码无需感知风控逻辑。拦截目标明确：发布帖子和主动发起私信。被动接收私信保留通道，不限制。

## 2. 技术栈

- **框架**: Spring Boot 4.x + Spring AOP
- **安全**: Spring Security + JWT
- **缓存**: Redis

## 3. 核心功能模块设计

### 3.1 风控规则定义

| 规则编号 | 触发条件 | 限制行为 | 错误提示 |
|:---------|:---------|:---------|:---------|
| R1 | credit_score < 60 | 发布帖文 | "您的信用分低于60分，当前已被限制发布权限，请规范交易行为" |
| R2 | credit_score < 60 | 主动发起私信 | "您的信用分低于60分，当前已被限制主动私信权限，请规范交易行为" |
| R3 | credit_score < 60 | 标记成交 | "您的信用分低于60分，当前已被限制交易权限" |

**不限制的行为**（保留沟通通道）：
- 浏览信息流和搜索（看帖）
- 被动回复私信（回复别人发来的消息）
- 查看个人信息和修改密码
- 提交评价（收到评价是对其过去行为的反馈）

### 3.2 风控拦截实现方案

#### 3.2.1 自定义注解 + AOP 切面（推荐）

**自定义注解**：
```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface CreditCheck {
    int minScore() default 60;        // 最低信用分要求
    String message() default "信用分不足，权限已被限制";
}
```

**AOP 切面处理**：
```java
@Aspect
@Component
public class CreditCheckAspect {

    @Autowired
    private StringRedisTemplate redisTemplate;

    @Before("@annotation(creditCheck)")
    public void checkCreditScore(JoinPoint joinPoint, CreditCheck creditCheck) {
        // 1. 从 SecurityContext 获取当前用户 ID
        String userId = SecurityContextHolder.getContext()
                .getAuthentication().getPrincipal().toString();

        // 2. 从 Redis 读取缓存的信用分（优先），缓存未命中则查 DB
        String cacheKey = "credit:score:" + userId;
        String cachedScore = redisTemplate.opsForValue().get(cacheKey);
        Integer creditScore;

        if (cachedScore != null) {
            creditScore = Integer.parseInt(cachedScore);
        } else {
            // 查询数据库并缓存
            creditScore = userMapper.findById(Long.valueOf(userId)).getCreditScore();
            redisTemplate.opsForValue().set(cacheKey, String.valueOf(creditScore), 5, TimeUnit.MINUTES);
        }

        // 3. 检查信用分是否满足最低要求
        if (creditScore < creditCheck.minScore()) {
            throw new BusinessException(403, creditCheck.message());
        }
    }
}
```

**使用方式**（在 Controller 方法上加注解）：
```java
@PostMapping("/publish")
@CreditCheck(minScore = 60, message = "您的信用分低于60分，当前已被限制发布权限，请规范交易行为")
public CommonResult<ItemPublishResp> publish(...) { ... }

@PostMapping("/session/init")
@CreditCheck(minScore = 60, message = "您的信用分低于60分，当前已被限制主动私信权限，请规范交易行为")
public CommonResult<SessionInitResp> initSession(...) { ... }
```

#### 3.2.2 方案对比

| 方案 | 优点 | 缺点 | 推荐 |
|:-----|:-----|:-----|:-----|
| **@Aspect 注解** | 声明式、低侵入、可复用 | 需要引入 AOP 依赖 | ✅ 推荐 |
| 业务代码硬编码 | 简单直接 | 分散在各 Controller，难维护 | 初期可用 |
| Spring Interceptor | 统一拦截，无需注解 | 需要维护 URL 路径映射 | 中等 |

**推荐方案**：项目初期在业务代码中直接校验（简单），随接口增多后迁移到 @CreditCheck 注解 + AOP。

### 3.3 信用分缓存策略

**业务目标**：每次风控检查都查数据库会造成不必要的 IO，需要缓存层加速。

#### 3.3.1 缓存设计

| Key 格式 | 类型 | 用途 | TTL | 更新策略 |
|:---------|:-----|:-----|:----|:---------|
| `credit:score:{userId}` | String | 信用分缓存 | 5 分钟 | 分数变化时主动失效 |

#### 3.3.2 缓存更新时机

| 触发事件 | 操作 |
|:---------|:-----|
| 评价提交 → 信用分重算完成 | `redisTemplate.delete("credit:score:" + userId)` |
| 放鸽子核实扣分 | 同上 |
| 缓存过期自然淘汰 | 下次请求时从 DB 加载 |

#### 3.3.3 缓存穿透防护
- 缓存中不存在时查询数据库
- 用户不存在时也缓存标记（如 `"NON_EXISTENT"`），TTL 1 分钟
- 信用分不变时，缓存自动过期后重新加载

### 3.4 信用分变更时的状态联动

**业务目标**：信用分跌破阈值时自动变更用户状态，分数恢复时自动解除限制。

#### 3.4.1 联动逻辑
```java
public void updateCreditScore(Long userId, int newScore) {
    // 更新信用分
    userMapper.updateCreditScore(userId, newScore);

    // 检查是否需要变更用户状态
    if (newScore < 60) {
        userMapper.updateUserStatus(userId, UserStatusEnum.BANNED.getCode());
        log.warn("用户 {} 信用分低于60，已自动限制权限", userId);
    } else if (newScore >= 60) {
        // 仅当之前是被禁用状态时才恢复
        User user = userMapper.findById(userId);
        if (user.getUserStatus() == UserStatusEnum.BANNED.getCode()) {
            userMapper.updateUserStatus(userId, UserStatusEnum.NORMAL.getCode());
            log.info("用户 {} 信用分恢复至60以上，已解除权限限制", userId);
        }
    }

    // 主动失效缓存
    redisTemplate.delete("credit:score:" + userId);
}
```

#### 3.4.2 状态流转

```
                注册（100分）
                    │
         ┌─────────┴─────────┐
         │                   │
    信用分 ≥ 60          信用分 < 60
    user_status=0        user_status=1
    （正常）              （禁用）
         │                   │
         │   信用分恢复 ≥ 60  │
         └───────────────────┘
```

## 5. 安全设计

### 5.1 不可绕过
- 风控检查在后端独立执行，不依赖前端传参
- 即使用户绕过前端直接调 API，后端也会拦截

### 5.2 最小权限
- 被动接收私信不限制（保留沟通通道）
- 浏览信息流不限制（权利与信用分离）

### 5.3 防并发
- 信用分更新使用原子 SQL 操作
- AOP 拦截器中读取的是缓存值，缓存由更新操作主动失效

## 7. 性能优化设计

### 7.1 缓存优化
- 信用分缓存 TTL 5 分钟，风控检查走 Redis（< 1ms）
- 缓存命中率预计 > 95%（信用分不频繁变更）

### 7.2 AOP 优化
- 风控切面仅拦截带 `@CreditCheck` 注解的方法，不影响其他接口性能
- 切面中只做 Redis 查询（极快），不走网络调用

|:-------|:-----|:-----|
