# 交易状态闭环与双向评价 - 后端技术设计文档

## 1. 概述

本文档详细描述校园交易平台交易状态闭环与双向评价的后端技术实现方案，包括标记成交节点、提交评价、评价标签管理等功能模块的技术架构、接口设计、数据模型和安全策略。

**核心设计原则**：极简交易流转——发布者标记成交并指定匹配用户，即可形成撮合闭环。双向互评机制确保双方信用档案的真实性。评价提交后即时生效，异步触发下一环节的信用分计算。

## 2. 技术栈

- **框架**: Spring Boot 4.x
- **安全**: Spring Security + JWT
- **数据库**: MySQL 8.0+
- **缓存**: Redis
- **事务**: Spring Transaction（`@Transactional`）
- **异步**: Spring Async（`@Async`）
- **验证**: Jakarta Validation

## 3. 核心功能模块设计

### 3.1 标记成交节点模块 (`/api/item/{id}/deal`)

**业务目标**：发布者确认交易完成，关联成交对象，触发双向评价通知，形成交易闭环。

#### 3.1.1 API 设计

##### 标记成交节点
- **接口**: `POST /api/item/{id}/deal`
- **请求头**: `Authorization: Bearer {token}`
- **路径参数**:
```
id: long  // 帖文ID
```
- **请求参数**:
```json
{
  "targetUserId": "long"  // 成交对象用户ID（必填，需为有私信记录的用户）
}
```
- **响应参数**:
```json
{
  "code": 200,
  "message": "标记成交成功，请前往评价",
  "data": {
    "itemId": "long",
    "evaluateDeadline": "datetime"  // 评价截止时间（+7天）
  }
}
```

#### 3.1.2 业务逻辑设计

##### 标记成交流程（有序，事务保障）
1. **归属权校验**：查询帖文，验证 `user_id` = 当前登录用户 ID
2. **状态校验**：帖文状态必须为"展示中"（`item_status = 0`）
3. **成交对象校验**：
   - `targetUserId` 对应的用户存在且账号状态正常
   - 发布者与成交对象之间存在私信记录（`message` 表有记录）
4. **事务执行**：
   - 更新 `item_status = 1`（已成交）
   - 写入 `matched_user_id = {targetUserId}`
   - 写入 `deal_at = NOW()`（成交时间）
5. **异步通知**（事务提交后）：
   - 向双方发送站内信，内容包含"去评价"的引流跳转信息
   - 站内信类型标记为"评价提醒"

##### 评价提醒站内信内容
- **发给发布者**："恭喜！你已确认与 {对方昵称} 完成交易，请对本次交易进行评价。评价截止：{deadline}。"
- **发给成交对象**："你的交易已被 {发布者昵称} 确认为成交，请对本次交易进行评价。评价截止：{deadline}。"

### 3.2 提交评价模块 (`/api/evaluate/submit`)

**业务目标**：用户提交对交易对方的评价（标签 + 文字），评价即时生效并触发信用分重算。

#### 3.2.1 API 设计

##### 提交评价
- **接口**: `POST /api/evaluate/submit`
- **请求头**: `Authorization: Bearer {token}`
- **请求参数**:
```json
{
  "itemId": "long",         // 关联帖文ID（必填）
  "revieweeId": "long",     // 被评价人ID（必填）
  "tags": "string",         // 评价标签，逗号分隔（必填，如"爽快,准时"）
  "content": "string",      // 补充文字，最长50字（可选）
  "isPositive": "boolean"   // true=好评, false=差评
}
```
- **响应参数**:
```json
{
  "code": 200,
  "message": "评价提交成功",
  "data": {
    "reviewId": "long",
    "creditChange": "integer"  // 被评价人信用分变化（如+2 / -2）供前端临时展示
  }
}
```

#### 3.2.2 业务逻辑设计

##### 提交评价流程
1. **权限校验**：
   - 该帖文 `item_status = 1`（已成交）
   - 当前用户为该帖文的发布者或成交对象（`matched_user_id`）
   - 评价人 ≠ 被评价人（不可自评）
2. **重复评价检查**：
   - 同一用户对同一帖文只能评价一次
   - `SELECT COUNT(1) FROM review WHERE item_id=? AND reviewer_id=?`
3. **评价期校验**（可选）：
   - 成交后 7 天内可评价，超期不可评价
4. **标签校验**：
   - 标签必须在系统预置的标签库内，防止自定义恶意标签
5. **写入评价**：
   ```sql
   INSERT INTO review (item_id, reviewer_id, reviewee_id, rating_type, tags, content)
   VALUES (?, ?, ?, ?, ?, ?)
   -- rating_type: 1=好评(+2), -1=差评(-2)
   ```
6. **异步触发信用分重算**（见模块 3 详细设计）

##### 评价标签库

| 评价性质 | rating_type | 分数影响 | 标签选项 |
|:---------|:------------|:---------|:---------|
| 好评 | 1 | +2 | 爽快、准时、描述一致、好卖家、好买家、有耐心、物超所值、沟通顺畅 |
| 差评 | -1 | -2 | 砍价墨迹、描述不符、放鸽子、态度差、物品瑕疵、迟迟不发货 |

## 4. 数据模型设计

### 4.1 评价表 (review)

已建表，核心字段回顾：

| 字段名 | 数据类型 | 说明 |
|:-------|:---------|:-----|
| `id` | BIGINT | 评价流水ID |
| `item_id` | BIGINT | 关联帖文ID |
| `reviewer_id` | BIGINT | 评价人ID |
| `reviewee_id` | BIGINT | 被评价人ID |
| `rating_type` | TINYINT | 1=好评(+2), 0=中评(0), -1=差评(-2) |
| `tags` | VARCHAR(100) | 结构化标签（逗号分隔） |
| `content` | VARCHAR(255) | 文字短评（限 50 字） |
| `created_at` | DATETIME | 评价时间 |

### 4.2 帖文表 (item)

成交相关字段：

| 字段名 | 数据类型 | 说明 |
|:-------|:---------|:-----|
| `item_status` | TINYINT | 0=展示中, 1=已成交 |
| `matched_user_id` | BIGINT | 成交对象ID |
| `deal_at` | DATETIME | 成交时间（评价截止时间 = deal_at + 7 天） |

### 4.3 Redis 数据结构设计

| Key 格式 | 类型 | 用途 | 过期时间 |
|:---------|:-----|:-----|:---------|
| `evaluate:deadline:{itemId}` | String | 评价截止时间戳 | 成交后 7 天 |

## 5. 安全设计

### 5.1 业务规则
- 只有帖文发布者和成交对象可以提交评价
- 同一用户对同一帖文只能评价一次
- 不可给自己评价
- 成交后 7 天为评价有效期，超期不可评价

### 5.2 数据完整性
- 标记成交使用事务，保证 `item_status` 和 `matched_user_id` 原子更新
- 评价通知发送在事务提交后执行，避免长事务
