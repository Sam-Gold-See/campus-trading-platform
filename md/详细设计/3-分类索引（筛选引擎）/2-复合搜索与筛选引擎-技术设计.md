# 复合搜索与筛选引擎 - 后端技术设计文档

## 1. 概述

本文档详细描述校园交易平台复合搜索与筛选引擎的后端技术实现方案。通过关键词模糊搜索 + 多维结构化筛选的组合查询，让用户在庞杂的信息流中迅速定位目标需求，提高撮合效率。

**核心设计原则**：灵活参数（全部非必填）、动态 SQL 拼接、排序降级策略、防爆破限制。所有搜索仅在"展示中"的有效帖文池内进行。

## 2. 技术栈

- **框架**: Spring Boot 4.x
- **安全**: Spring Security + JWT
- **数据库**: MySQL 8.0+（利用 FULLTEXT 全文索引）
- **验证**: Jakarta Validation

## 3. 核心功能模块设计

### 3.1 综合信息检索模块 (`/api/item/search`)

#### 3.1.1 API 设计

##### 综合信息检索
- **接口**: `GET /api/item/search`
- **请求头**: `Authorization: Bearer {token}`
- **请求参数**（全部非必填）:
```
keyword: string      // 搜索关键词（最长20字）
type: integer        // 1=求购, 2=转让（不传则查所有类型）
categoryId: integer  // 分类ID
campus: string       // 校区标识
priceRange: string   // 价格区间枚举标识
                     // free(面议) / under50(50元以下) / 50-200 / 200-500 / over500(500元以上)
page: integer        // 页码（从1开始，默认1）
size: integer        // 每页条数（默认20，最大50）
```
- **响应参数**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "list": [
      {
        "id": "long",
        "type": "integer",
        "categoryName": "string",
        "campus": "string",
        "price": "decimal",
        "content": "string",
        "imageUrl": "string",
        "publisherNickname": "string",
        "publisherCreditScore": "integer",
        "createdAt": "datetime",
        "relevanceScore": "double"  // 仅关键词搜索时有值，用于相关度排序
      }
    ],
    "total": "long",
    "page": "integer",
    "size": "integer"
  }
}
```

#### 3.1.2 业务逻辑设计

##### 查询构建流程
1. **基础池限制**（强制条件）：
   ```sql
   WHERE item_status = 0 AND expire_at > NOW()
   ```
   所有查询必须在有效帖文池内进行，屏蔽已成交、已失效、已过期的数据。

2. **结构化字段拼接**（AND 条件，直接命中索引）：
   ```sql
   -- 仅当参数非空时拼接
   AND type = #{type}                    -- 交易类型
   AND category_id = #{categoryId}       -- 分类ID
   AND campus = #{campus}                -- 校区
   ```

3. **价格区间处理**（AND 条件）：
   ```sql
   -- 根据 priceRange 参数动态拼接
   'free'     → AND price IS NULL
   'under50'  → AND price < 50
   '50-200'   → AND price >= 50 AND price <= 200
   '200-500'  → AND price >= 200 AND price <= 500
   'over500'  → AND price > 500
   -- 不传该参数 → 不拼接价格条件
   ```

4. **关键词搜索**（FULLTEXT 或 LIKE）：
   - **初期方案**（数据量 < 1 万条）：
     ```sql
     AND content LIKE CONCAT('%', #{keyword}, '%')
     ```
     简单直接，无需额外配置。

   - **进阶方案**（MySQL 5.7+，利用已有 FULLTEXT 索引）：
     ```sql
     AND MATCH(content) AGAINST(#{keyword} IN BOOLEAN MODE)
     ```
     支持中文分词、布尔操作符（+ - * ""），搜索效率远超 LIKE。

   - **切换策略**：通过配置项控制，初期用 LIKE，数据量增长后一键切换 FULLTEXT。

5. **排序降级策略**：
   ```
   有关键词 → ORDER BY MATCH(content) AGAINST(...) DESC  -- 按相关度
               或 ORDER BY created_at DESC                -- LIKE 模式下降级为时间排序
   无关键词 → ORDER BY created_at DESC                  -- 按最新/擦亮时间
   ```

##### Dynamic SQL 伪代码（MyBatis）
```xml
<select id="search" resultType="ItemSearchResult">
    SELECT i.*, u.nickname, u.credit_score, c.name AS categoryName
    <if test="keyword != null and keyword != ''">
        , MATCH(i.content) AGAINST(#{keyword} IN BOOLEAN MODE) AS relevanceScore
    </if>
    FROM item i
    JOIN user u ON i.user_id = u.id
    JOIN category c ON i.category_id = c.id
    WHERE i.item_status = 0
      AND i.expire_at > NOW()
    <if test="type != null">
        AND i.type = #{type}
    </if>
    <if test="categoryId != null">
        AND i.category_id = #{categoryId}
    </if>
    <if test="campus != null and campus != ''">
        AND i.campus = #{campus}
    </if>
    <if test="priceRange != null">
        <choose>
            <when test="priceRange == 'free'">AND i.price IS NULL</when>
            <when test="priceRange == 'under50'">AND i.price &lt; 50</when>
            <when test="priceRange == '50-200'">AND i.price &gt;= 50 AND i.price &lt;= 200</when>
            <when test="priceRange == '200-500'">AND i.price &gt;= 200 AND i.price &lt;= 500</when>
            <when test="priceRange == 'over500'">AND i.price &gt; 500</when>
        </choose>
    </if>
    <if test="keyword != null and keyword != ''">
        AND MATCH(i.content) AGAINST(#{keyword} IN BOOLEAN MODE)
    </if>
    <choose>
        <when test="keyword != null and keyword != ''">
            ORDER BY relevanceScore DESC
        </when>
        <otherwise>
            ORDER BY i.created_at DESC
        </otherwise>
    </choose>
    LIMIT #{offset}, #{size}
</select>
```

##### 防爆破设计
- **关键词长度限制**：最长 20 字，超长字符串直接拒绝（防止耗尽数据库性能）
- **特殊字符过滤**：去除 SQL 通配符 `% _` 等，防止 LIKE 注入
- **分页上限**：size 最大 50，防止单次拉取过多数据

### 3.2 价格区间枚举定义

| 枚举值 | 含义 | SQL 条件 |
|:-------|:-----|:---------|
| `free` | 面议 | `price IS NULL` |
| `under50` | 50元以下 | `price < 50` |
| `50-200` | 50-200元 | `price >= 50 AND price <= 200` |
| `200-500` | 200-500元 | `price >= 200 AND price <= 500` |
| `over500` | 500元以上 | `price > 500` |

## 4. 数据模型设计

### 4.1 涉及的数据库表

| 表 | 角色 | 关键字段 |
|:---|:-----|:---------|
| `item` | 主搜索表 | id, type, category_id, campus, price, content, item_status, created_at, expire_at |
| `user` | 发布者信息 | nickname, credit_score |
| `category` | 分类名称 | name |

### 4.2 索引利用

| 索引 | 用途 | 命中场景 |
|:-----|:-----|:---------|
| `idx_search (item_status, type, category_id, campus)` | 多维筛选 | 无关键词的条件组合查询 |
| `FULLTEXT(content)` | 全文搜索 | 有关键词时的文本匹配 |

**索引命中示例**：
```sql
-- 搜索"转让 + 书籍教材 + 南校区" → 完全命中 idx_search 前三列
WHERE item_status=0 AND type=2 AND category_id=1 AND campus='南校区'
```

## 5. 安全设计

### 5.1 认证授权
- 搜索接口需 JWT 认证（确保只有校内用户可搜索）
- 不暴露帖文发布者的敏感信息（仅返回昵称、信用分）

### 5.2 防注入设计
- **SQL 注入防护**：MyBatis 参数化查询（`#{}`），不使用 `${}` 拼接用户输入
- **LIKE 通配符过滤**：对用户输入的关键词，去除 `%` `_` 等 MySQL 通配符
- **关键词长度限制**：应用层强制 ≤20 字

### 5.3 数据防腐
- 强制 `item_status = 0`，不信任前端传入的任何状态参数
- 强制 `expire_at > NOW()`，过期帖不可搜索

## 7. 性能优化设计

### 7.1 数据库优化
- **联合索引 `idx_search`**：最左前缀匹配，无关键词场景下直接命中
- **全文索引 `FULLTEXT(content)`**：MySQL 内置分词引擎，比 `LIKE '%x%'` 快 10-100 倍
- **避免全表扫描**：至少有一个结构化条件（type/categoryId/campus）时能命中索引
- **分页限制**：单次最多 50 条，防止大 offset 深分页

### 7.2 LIKE vs FULLTEXT 对比

| 维度 | LIKE '%keyword%' | FULLTEXT |
|:-----|:-----------------|:---------|
| 索引利用 | 无法使用索引 | 使用全文索引 |
| 性能 | 全表扫描，数据量大时极慢 | 分词匹配，毫秒级 |
| 中文支持 | 原生支持 | MySQL 5.7+ ngram 分词器支持 |
| 相关度排序 | 不支持 | 支持 |
| 配置成本 | 零 | 需已有 FULLTEXT 索引（已建好） |

### 7.3 缓存策略（可选）
- 热门搜索词的结果缓存（Redis，TTL 30 秒）
- 搜索结果为空时也缓存（TTL 60 秒），防止缓存穿透

## 8. 与信息流接口的关系

| 接口 | 定位 | 区别 |
|:-----|:-----|:-----|
| `GET /api/item/feed` | 首页信息流 | 固定 type，无筛选，纯浏览 |
| `GET /api/item/search` | 搜索筛选 | 灵活参数，多维度组合查询 |

两者底层查询逻辑可复用同一套查询构建器，通过 `type` 参数区分"必填"与"可选"。

## 9. 实施优先级建议

| 优先级 | 功能 | 理由 |
|:-------|:-----|:-----|
| P0 | GET /api/item/search（基础版：结构化筛选 + LIKE） | 先跑通流程 |
