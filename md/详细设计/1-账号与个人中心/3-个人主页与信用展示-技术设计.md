# 个人主页与信用展示 - 后端技术设计文档

## 1. 概述

本文档详细描述校园交易平台个人主页与信用展示的后端技术实现方案，包括用户个人信息获取、隐私脱敏、信用分展示和评价列表查询等功能模块的技术架构、接口设计、数据模型和安全策略。

**核心设计原则**：通过信用分和历史评价建立交易对象的信任感，后端负责数据脱敏和结构化输出。

## 2. 技术栈

- **框架**: Spring Boot 4.x
- **安全**: Spring Security + JWT
- **数据库**: MySQL 8.0+
- **缓存**: Redis
- **验证**: Jakarta Validation

## 3. 核心功能模块设计

### 3.1 获取个人信息模块 (`/api/user/profile`)

#### 3.1.1 API 设计

##### 获取个人信息
- **接口**: `GET /api/user/profile`
- **请求头**: `Authorization: Bearer {token}`
- **响应参数**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "id": "long",
    "eduEmail": "string",
    "nickname": "string",
    "avatarUrl": "string",
    "creditScore": "integer",
    "userStatus": "integer",
    "isAdmin": "integer",
    "createdAt": "datetime"
  }
}
```

#### 3.1.2 业务逻辑设计

##### 获取个人信息流程
1. 从 JWT Token 中解析当前用户 ID
2. 根据 ID 查询 `user` 表获取用户信息
3. **隐私脱敏逻辑**：
   - 教育邮箱部分脱敏："test2026@m.scnu.edu.cn" → "t***6@m.scnu.edu.cn"
   - 若系统扩展了真实姓名字段，需执行"张*丰"式脱敏后下发
4. **信用分徽章映射**（后端计算建议值下发，前端渲染对应颜色）：
   - `<60 分` → 红色警告（限制发布/私信权限）
   - `60-80 分` → 灰色（普通用户）
   - `80-100 分` → 蓝色（良好用户）
   - `>100 分` → 金色（优秀用户）
5. 封装 `UserProfileResp` 返回，**禁止返回 password_hash**

### 3.2 获取用户评价列表模块 (`/api/user/reviews`)

#### 3.2.1 API 设计

##### 获取用户评价列表
- **接口**: `GET /api/user/reviews`
- **请求头**: `Authorization: Bearer {token}`
- **请求参数**:
```
targetUserId: long  // 要查看的用户ID（可选，默认当前登录用户）
page: integer       // 页码（从1开始，默认1）
size: integer       // 每页条数（默认10，最大50）
```
- **响应参数**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "list": [
      {
        "id": "long",
        "itemId": "long",
        "ratingType": "integer",    // 1=好评, 0=中评, -1=差评
        "tags": "string",           // 结构化标签，逗号分隔
        "content": "string",        // 文字短评
        "reviewerNickname": "string", // 评价人昵称（脱敏）
        "createdAt": "datetime"
      }
    ],
    "total": "long",
    "page": "integer",
    "size": "integer",
    // 评价统计
    "statistics": {
      "goodCount": "long",    // 好评数
      "neutralCount": "long", // 中评数
      "badCount": "long"     // 差评数
    }
  }
}
```

#### 3.2.2 业务逻辑设计

##### 评价列表查询流程
1. 获取 `targetUserId` 参数，默认取当前登录用户 ID
2. 查询 `review` 表，条件：`reviewee_id = targetUserId`
3. 按 `created_at` 倒序排列，支持分页
4. 联表查询评价人昵称（`user` 表），对昵称做基础脱敏
5. **隐私保护**：不返回 `reviewer_id` 的明文 ID，仅展示脱敏后的昵称
6. 额外返回评价统计（好评/中评/差评计数）

##### 评价标签枚举（系统预置）

| 类型 | 标签 |
|:-----|:-----|
| 正向 | 爽快、准时、描述一致、好卖家、好买家、有耐心 |
| 负向 | 砍价墨迹、描述不符、放鸽子、态度差、物品有瑕疵 |
| 中性 | 一般、还行 |

## 4. 数据模型设计

### 4.1 用户表 (user)

信用分相关字段：

| 字段名 | 数据类型 | 说明 |
|:-------|:---------|:-----|
| `credit_score` | INT | 信用分，默认100 |
| `user_status` | TINYINT | 0=正常, 1=禁用（信用分<60时系统自动变更） |

### 4.2 评价表 (review)

已建表，核心字段回顾：

| 字段名 | 数据类型 | 说明 |
|:-------|:---------|:-----|
| `id` | BIGINT | 评价流水ID |
| `item_id` | BIGINT | 关联的交易帖文ID |
| `reviewer_id` | BIGINT | 评价人ID |
| `reviewee_id` | BIGINT | 被评价人ID |
| `rating_type` | TINYINT | 1=好评, 0=中评, -1=差评 |
| `tags` | VARCHAR(100) | 结构化标签（逗号分隔） |
| `content` | VARCHAR(255) | 文字短评 |
| `created_at` | DATETIME | 评价时间 |

**核心索引**：
- `idx_reviewee (reviewee_id)` — 按被评价人查询评价列表
- `idx_item (item_id)` — 按帖文查询关联评价

## 5. 安全设计

### 5.1 隐私保护
- 邮箱脱敏：保留首字符和域名，中间字符替换为 `***`
- 评价列表不暴露评价人真实 ID，仅展示脱敏后的昵称
- `password_hash` 绝不通过任何接口返回

### 5.2 认证授权
- 所有接口均需 JWT 认证
- 查看他人评价列表时不做权限限制（评价是公开信息，用于建立信任）

---

## 6. 异常处理设计

### 6.1 统一异常处理
复用现有 `GlobalExceptionHandler`：

- `BusinessException`: 业务异常（如"用户不存在"）
- `AuthenticationException`: 认证异常（无 Token 或 Token 失效）

### 6.2 错误码设计

```java
USER_NOT_FOUND(40402, "用户不存在"),
```
