# 我的发布与状态管理 - 后端技术设计文档

## 1. 概述

本文档详细描述校园交易平台"我的发布与状态管理"的后端技术实现方案，包括帖文列表查询（按状态筛选）、擦亮置顶、提前下架、标记成交等功能模块的技术架构、接口设计、数据模型和安全策略。

**核心设计原则**：所有操作均需校验帖文归属权（user_id 匹配），防止越权操作他人的帖文。

## 2. 技术栈

- **框架**: Spring Boot 4.x
- **安全**: Spring Security + JWT
- **数据库**: MySQL 8.0+
- **缓存**: Redis
- **事务**: Spring Transaction (`@Transactional`)
- **验证**: Jakarta Validation

## 3. 核心功能模块设计

### 3.1 我的发布列表模块 (`/api/item/my-list`)

**业务目标**：允许用户查看自己发布的全部帖文，按状态 Tab 切换（进行中、已完成、已失效）。

#### 3.1.1 API 设计

##### 查询我的发布
- **接口**: `GET /api/item/my-list`
- **请求头**: `Authorization: Bearer {token}`
- **请求参数**:
```
status: integer  // 0=进行中, 1=已完成, 2=已失效（默认0）
page: integer    // 页码（从1开始，默认1）
size: integer    // 每页条数（默认20，最大50）
```
- **响应参数**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "list": [
      {
        "id": "long",
        "type": "integer",
        "categoryName": "string",
        "campus": "string",
        "price": "decimal",
        "content": "string",
        "imageUrl": "string",
        "itemStatus": "integer",
        "matchedUserId": "long",
        "createdAt": "datetime",
        "expireAt": "datetime"
      }
    ],
    "total": "long",
    "page": "integer",
    "size": "integer"
  }
}
```

#### 3.1.2 业务逻辑设计

1. 从 JWT Token 获取当前用户 ID
2. 查询 `item` 表：`user_id = 当前用户 AND item_status = {入参status}`
3. 按 `created_at` 降序排列（最新在前）
4. 联表查询分类名称（`category` 表）
5. 分页返回

### 3.2 帖文擦亮模块 (`/api/item/{id}/bump`)

**业务目标**：将指定帖文升到信息流顶部，每天限制 1 次。

#### 3.2.1 API 设计

##### 帖文擦亮
- **接口**: `PUT /api/item/{id}/bump`
- **请求头**: `Authorization: Bearer {token}`
- **路径参数**:
```
id: long  // 帖文ID
```
- **响应参数**:
```json
{
  "code": 200,
  "message": "擦亮成功"
}
```

#### 3.2.2 业务逻辑设计

##### 擦亮流程
1. **归属权校验**：查询帖文，验证 `user_id = 当前用户 ID`，否则抛出 `ITEM_NOT_OWNER`
2. **状态校验**：帖文状态必须为"展示中"（`item_status = 0`），否则抛出 `ITEM_STATUS_INVALID`
3. **每日限制**：Redis 检查 `bump:{itemId}:{yyyyMMdd}` 是否存在
   - 存在 → 抛出 `BUMP_LIMIT_EXCEEDED`（今日已达到擦亮次数上限）
   - 不存在 → 继续
4. **执行擦亮**：更新 `created_at = NOW()`（利用现有排序机制实现"顶帖"效果）
5. **设置标记**：Redis 写入 `bump:{itemId}:{yyyyMMdd}` = "1"，过期时间为次日 00:00

##### 擦亮次数限制设计

| Key 格式 | 类型 | 用途 | 过期策略 |
|:---------|:-----|:-----|:---------|
| `bump:{itemId}:{yyyyMMdd}` | String | 该帖今日是否已擦亮 | 次日 00:00 自动过期 |

**计算过期秒数**：`(明天零点 - 当前时间)` 的秒数差，确保每天零点自动重置。

### 3.3 提前下架模块 (`/api/item/{id}/offline`)

**业务目标**：将正在展示的帖文提前下架，移出需求墙展示池。

#### 3.3.1 API 设计

##### 提前下架
- **接口**: `PUT /api/item/{id}/offline`
- **请求头**: `Authorization: Bearer {token}`
- **路径参数**:
```
id: long  // 帖文ID
```
- **响应参数**:
```json
{
  "code": 200,
  "message": "下架成功"
}
```

#### 3.3.2 业务逻辑设计

##### 下架流程
1. **归属权校验**：查询帖文，验证 `user_id = 当前用户 ID`
2. **状态校验**：帖文状态必须为"展示中"（`item_status = 0`），已下架或已成交的帖文不可重复操作
3. **执行下架**：更新 `item_status = 2`（已失效）
4. 无需事务（单表单字段更新）

---

### 3.4 标记成交模块 (`/api/item/{id}/sold`)

**业务目标**：发布者确认交易完成，关联成交对象，触发评价通知。

#### 3.4.1 API 设计

##### 标记成交
- **接口**: `PUT /api/item/{id}/sold`
- **请求头**: `Authorization: Bearer {token}`
- **路径参数**:
```
id: long  // 帖文ID
```
- **请求参数**:
```json
{
  "matchedUserId": "long"  // 成交对象用户ID（必填，需为有私信记录的用户）
}
```
- **响应参数**:
```json
{
  "code": 200,
  "message": "标记成交成功"
}
```

#### 3.4.2 业务逻辑设计

##### 标记成交流程
1. **归属权校验**：查询帖文，验证 `user_id = 当前用户 ID`
2. **状态校验**：帖文状态必须为"展示中"（`item_status = 0`）
3. **成交对象校验**：验证 `matchedUserId` 对应的用户存在且状态正常
4. **私信记录校验**（可选）：检查发布者与成交对象之间是否存在私信记录（`message` 表），防止虚假成交
5. **事务执行**（`@Transactional`）：
   - 更新 `item_status = 1`（已成交）
   - 写入 `matched_user_id = {成交对象ID}`
6. **异步通知**（事务提交后）：
   - 向成交对象发送站内信："恭喜！{发布者昵称} 已确认与你完成交易，请前往评价"
   - 向发布者发送站内信："请对 {成交对象昵称} 进行评价"
   - 消息类型标记为"评价提醒"，附带跳转评价页面的链接参数

##### 事务设计

| 步骤 | 操作 | 说明 |
|:-----|:-----|:-----|
| 1 | `UPDATE item SET item_status=1, matched_user_id=? WHERE id=?` | 原子更新 |
| 2 | `INSERT INTO message (...) VALUES (...)` | 写入系统通知（事务内，保证一致性） |
| 3 | 提交事务 | |
| 4 | 异步推送 WebSocket 通知 | 事务外，避免长连接阻塞 DB 事务 |

## 4. 数据模型设计

### 4.1 需求/物品表 (item)

状态管理相关字段：

| 字段名 | 数据类型 | 说明 |
|:-------|:---------|:-----|
| `id` | BIGINT | 帖文ID |
| `user_id` | BIGINT | 发布者ID（归属权校验依据） |
| `item_status` | TINYINT | 0=展示中, 1=已成交, 2=已失效 |
| `matched_user_id` | BIGINT | 成交方用户ID（标记成交时写入） |
| `created_at` | DATETIME | 发布时间（擦亮=更新此字段） |
| `expire_at` | DATETIME | 过期时间（+14天） |

**状态流转图**：
```
0(展示中) ──标记成交──→ 1(已成交)
0(展示中) ──提前下架──→ 2(已失效)
0(展示中) ──14天过期──→ 2(已失效)
```

### 4.2 站内信表 (message)

系统通知消息（评价提醒）写入此表：

| 字段名 | 数据类型 | 说明 |
|:-------|:---------|:-----|
| `id` | BIGINT | 消息ID |
| `sender_id` | BIGINT | 发送方ID（系统通知用 0 表示系统） |
| `receiver_id` | BIGINT | 接收方ID |
| `item_id` | BIGINT | 关联帖文ID |
| `content` | VARCHAR(500) | 通知内容 |
| `is_read` | TINYINT | 0=未读, 1=已读 |

### 4.3 Redis 数据结构设计

| Key 格式 | 类型 | 用途 | 过期时间 |
|:---------|:-----|:-----|:---------|
| `bump:{itemId}:{yyyyMMdd}` | String | 帖文当日擦亮标记 | 次日 00:00 |

## 5. 安全设计

### 5.1 归属权校验
- 所有帖文写操作（擦亮/下架/标记成交）必须校验 `item.user_id = 当前登录用户ID`
- 校验逻辑抽取为公共方法，避免遗漏

### 5.2 状态机保护
- 操作前校验帖文当前状态是否允许该操作
- 仅"展示中"（status=0）的帖文可执行擦亮/下架/标记成交
- 已成交、已失效的帖文不可重复操作

### 5.3 认证授权
- 所有接口均需 JWT 认证
- 擦亮/下架/标记成交需额外校验归属权

---

## 6. 异常处理设计

### 6.1 统一异常处理
复用现有 `GlobalExceptionHandler`：

- `BusinessException`: 业务异常
- `AccessDeniedException`: 权限异常（操作他人帖文）

### 6.2 错误码设计

```java
ITEM_NOT_FOUND(40401, "帖文不存在"),
ITEM_NOT_OWNER(40302, "无权操作该帖文"),
ITEM_STATUS_INVALID(40001, "帖文状态不允许此操作"),
BUMP_LIMIT_EXCEEDED(42901, "今日已达到擦亮次数上限"),
MATCHED_USER_NOT_FOUND(40403, "成交对象不存在"),
NO_CHAT_RECORD(40002, "与成交对象无沟通记录，无法标记成交"),
```

## 7. 性能优化设计

### 7.1 数据库优化
- `idx_user_id (user_id)` 索引加速"我的发布"查询
- 擦亮操作仅更新 `created_at` 字段，避免全表更新

### 7.2 缓存策略
- 每日擦亮标记存 Redis，避免额外的数据库查询

## 8. 实施优先级建议

| 优先级 | 功能 | 理由 |
|:-------|:-----|:-----|
| P0 | GET /api/item/my-list | 前端"我的发布"页面依赖 |
| P0 | PUT /api/item/{id}/bump | 擦亮是高频用户操作 |
| P0 | PUT /api/item/{id}/offline | 帖文管理基础操作 |
| P1 | PUT /api/item/{id}/sold | 依赖站内信系统（模块4）通知功能 |
