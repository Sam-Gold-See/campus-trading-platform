# 长连接与实时收发引擎 - 后端技术设计文档

## 1. 概述

本文档详细描述校园交易平台 WebSocket 长连接与实时消息收发引擎的后端技术实现方案，包括 WebSocket 服务端网关、连接鉴权、连接池管理、消息双写（DB + 在线投递）、心跳机制和在线状态广播等功能模块。

**核心设计原则**：低延迟、不丢包、可感知在线状态。消息先落盘（持久化）再投递（实时性），在线直接推送，离线由 HTTP 补偿接口拉取。

## 2. 技术栈

- **框架**: Spring Boot 4.x + Spring WebSocket
- **安全**: Spring Security + JWT
- **数据库**: MySQL 8.0+
- **缓存**: Redis（Pub/Sub 跨节点通信）
- **连接池**: ConcurrentHashMap（单节点）、Redis Pub/Sub（多节点扩展）

## 3. 核心功能模块设计

### 3.1 WebSocket 服务端网关

**业务目标**：在连接握手阶段校验 JWT Token，建立并维护用户与 WebSocket Channel 的映射关系。

#### 3.1.1 连接鉴权

##### 鉴权流程
1. 客户端发起 WebSocket 连接请求：`ws://localhost:8080/ws/chat?token={JWT}`
2. 服务端在握手拦截器 `HandshakeInterceptor` 中：
   - 从 URL 参数 `token` 中提取 JWT
   - 验证 JWT 有效性和黑名单（复用 `JwtTokenProvider` 和 Redis）
   - 验证通过 → 允许握手，将 userId 存入 WebSocket Session attributes
   - 验证失败 → 拒绝连接，返回 401

##### 拦截器配置
```java
@Override
public void registerWebSocketHandlers(WebSocketHandlerRegistry registry) {
    registry.addHandler(chatWebSocketHandler, "/ws/chat")
            .addInterceptors(new JwtHandshakeInterceptor())
            .setAllowedOrigins("*");
}
```

#### 3.1.2 连接池管理

**数据结构**：
```java
// userId → WebSocket Session
private static final ConcurrentHashMap<Long, WebSocketSession> ONLINE_USERS = new ConcurrentHashMap<>();
```

**生命周期**：
- **建立连接**：握手成功后，`ONLINE_USERS.put(userId, session)`
- **断开连接**：`afterConnectionClosed` 触发，`ONLINE_USERS.remove(userId)`
- **上线广播**：连接建立后，通知对方的活跃会话列表"该用户已上线"
- **下线广播**：连接断开后，通知对方的活跃会话列表"该用户已离线"

##### 多节点扩展设计
单节点部署时使用 `ConcurrentHashMap` 即可。多节点部署时需借助 Redis Pub/Sub 实现跨节点消息路由：

```
节点A 收到消息 → 查 ONLINE_USERS，接收方不在A → 通过 Redis Pub/Sub 广播到节点B → 节点B 查找本地 ONLINE_USERS → 投递
```

### 3.2 消息实时收发引擎

**业务目标**：接收客户端文字消息，双写策略（先落盘 DB，再在线投递），确保消息不丢包。

#### 3.2.1 消息协议设计

##### 客户端 → 服务端（发送消息）
```json
{
  "type": "chat_message",
  "data": {
    "targetUserId": "long",
    "itemId": "long",
    "content": "string"     // 消息内容，最长 500 字
  }
}
```

##### 服务端 → 客户端（推送消息）
```json
{
  "type": "chat_message",
  "data": {
    "id": "long",
    "senderId": "long",
    "senderNickname": "string",
    "itemId": "long",
    "content": "string",
    "createdAt": "datetime"
  }
}
```

##### 服务端 → 客户端（在线状态变更）
```json
{
  "type": "online_status",
  "data": {
    "userId": "long",
    "online": "boolean"
  }
}
```

##### 客户端 → 服务端（心跳 Ping）
```json
{
  "type": "ping"
}
```

##### 服务端 → 客户端（心跳 Pong）
```json
{
  "type": "pong"
}
```

#### 3.2.2 消息处理流程

##### 发送消息完整流程
```
clientA ─────→ WS Server
                  │
                  │ 1. 解析消息包（targetUserId, itemId, content）
                  │ 2. 校验：targetUserId 存在、content 非空≤500字
                  │ 3. 风控：sender 信用分<60 可被动接收但不可主动发送
                  │
                  │ 4. 异步落盘到 message 表
                  │    INSERT INTO message (sender_id, receiver_id, item_id, content)
                  │    VALUES (?, ?, ?, ?)
                  │
                  │ 5. 检查接收方是否在线
                  │    ├─ 在线 → 直接通过 WS 推送消息
                  │    └─ 离线 → 跳过（下次登录时通过 HTTP 补偿拉取）
                  │
                  │ 6. 向发送方回执（消息ID + 时间戳）
```

##### 双写策略说明

| 顺序 | 操作 | 原因 |
|:-----|:-----|:-----|
| 先 | 落盘 DB | 持久化优先，确保消息不丢失 |
| 后 | 在线投递 | 在线则实时推送，离线则静默跳过 |

**注意**：不在线投递环节阻塞消息持久化。落盘成功后消息已"安全"，投递失败不影响数据完整性。

#### 3.2.3 心跳机制

- **客户端**：每 30 秒发送 `{ "type": "ping" }` 包
- **服务端**：收到 ping 立即回复 `{ "type": "pong" }`
- **断线检测**：服务端 90 秒未收到任何消息（含 ping），主动关闭连接
- **前端重连**：使用指数退避策略（1s → 2s → 4s → 8s → 16s → 最大 30s）

### 3.3 在线状态广播

**业务目标**：用户上下线时通知对方，前端可展示"当前在线/离线"状态。

#### 3.3.1 广播规则
1. 用户上线时：
   - 查询该用户的活跃会话对方（最近 7 天有消息往来的用户）
   - 遍历这些用户，对在线的推送上线通知
2. 用户离线时：
   - 同样遍历活跃会话对方
   - 对在线的推送离线通知

#### 3.3.2 活跃会话对方查询
```sql
SELECT DISTINCT
    CASE WHEN sender_id = #{userId} THEN receiver_id ELSE sender_id END AS contact_id
FROM message
WHERE (sender_id = #{userId} OR receiver_id = #{userId})
  AND created_at > DATE_SUB(NOW(), INTERVAL 7 DAY)
```

## 4. 数据模型设计

### 4.1 站内信表 (message)

已建表，消息收发核心字段：

| 字段名 | 数据类型 | 默认值 | 说明 |
|:-------|:---------|:-------|:-----|
| `id` | BIGINT | AUTO_INC | 消息ID（回执给发送方） |
| `sender_id` | BIGINT | — | 发送方ID |
| `receiver_id` | BIGINT | — | 接收方ID |
| `item_id` | BIGINT | — | 关联帖文ID |
| `content` | VARCHAR(500) | — | 消息内容 |
| `is_read` | TINYINT | 0 | 0=未读 |
| `created_at` | DATETIME | NOW() | 发送时间 |

### 4.2 连接池（内存）

| 数据结构 | Key | Value | 说明 |
|:---------|:----|:------|:-----|
| `ConcurrentHashMap` | userId (Long) | WebSocketSession | 在线用户连接池 |
| `ConcurrentHashMap` | sessionId (String) | userId (Long) | Session→用户反向映射 |

### 4.3 Redis 数据结构设计

| Key 格式 | 类型 | 用途 | 过期时间 |
|:---------|:-----|:-----|:---------|
| `ws:online:{userId}` | String | 用户在线状态（多节点共享） | 30s（心跳续期） |

## 5. 安全设计

### 5.1 连接层安全
- WebSocket 握手阶段 JWT 校验，无效 Token 拒绝连接
- 连接建立后仍需校验每条消息的 sender_id 与 JWT 中的 userId 一致

### 5.2 消息层安全
- 消息内容长度限制 500 字（与数据库 VARCHAR(500) 一致）
- 违规词过滤（复用发布模块的 Trie 树词库）
- 信用分 < 60 的用户：可被动接收消息，不可主动发送

### 5.3 防滥用
- 单用户最大连接数限制 3 个（防止恶意建立大量连接）
- 消息发送频率限制：单用户每秒最多 5 条（Redis 滑动窗口计数）

## 6. 异常处理设计

### 6.1 连接异常
- 非法 Token → 握手阶段拒绝，返回 HTTP 401
- 连接中断 → 清理连接池，广播离线状态
- 心跳超时 → 服务端主动断开，清理连接池

### 6.2 消息异常
- 消息落盘失败 → 重试 3 次，仍失败返回错误告知发送方
- 在线投递失败 → 静默跳过（消息已持久化，可通过离线补偿拉取）

### 6.3 错误码设计

```java
WEBSOCKET_UNAUTHORIZED(40101, "WebSocket连接鉴权失败"),
MESSAGE_RATE_LIMITED(42903, "消息发送过于频繁，请稍后再试"),
CONNECTION_LIMIT_EXCEEDED(42904, "连接数超限"),
```

## 7. 性能优化设计

### 7.1 连接池优化
- `ConcurrentHashMap` 保证线程安全和高并发
- 单节点支持 5000+ 并发连接

### 7.2 消息落盘优化
- 消息持久化使用异步方式（`@Async`），不阻塞 WS 线程
- 批量写入优化（可选，积攒多条消息后批量 INSERT）

### 7.3 心跳优化
- 心跳包极小（仅几个字节），30 秒一次，带宽消耗可忽略
- 心跳超时设为 90 秒（3 个心跳周期），避免网络抖动误断

## 8. 实施优先级建议

| 优先级 | 功能 | 理由 |
|:-------|:-----|:-----|
| P0 | WebSocket 网关 + 鉴权 | 基础连接能力 |
| P0 | 消息收发（双写策略） | 核心功能 |
| P0 | 连接池管理 | 消息投递依赖 |
| P1 | 心跳机制 | 稳定性保障 |
| P1 | 在线状态广播 | 体验增强 |
| P2 | 多节点 Redis Pub/Sub | 初期单节点无需 |
