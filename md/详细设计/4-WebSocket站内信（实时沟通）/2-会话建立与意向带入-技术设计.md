# 会话建立与意向带入 - 后端技术设计文档

## 1. 概述

本文档详细描述校园交易平台站内信会话建立与意向带入的后端技术实现方案，包括会话初始化（含商品意向卡片数据）、快捷短语获取等功能模块的技术架构、接口设计、数据模型和安全策略。

**核心设计原则**：降低破冰沟通成本——在会话创建时自动带入商品上下文信息，配合快捷短语，让买卖双方无需重复介绍物品就能直接切入交易沟通。

## 2. 技术栈

- **框架**: Spring Boot 4.x
- **安全**: Spring Security + JWT
- **数据库**: MySQL 8.0+
- **缓存**: Redis
- **验证**: Jakarta Validation

## 3. 核心功能模块设计

### 3.1 会话初始化模块 (`/api/chat/session/init`)

**业务目标**：从需求墙点击"聊一聊"后创建或获取会话，返回会话 ID、双方信息及商品意向卡片数据。

#### 3.1.1 API 设计

##### 初始化会话
- **接口**: `POST /api/chat/session/init`
- **请求头**: `Authorization: Bearer {token}`
- **请求参数**:
```json
{
  "targetUserId": "long",  // 对方用户ID（必填）
  "itemId": "long"         // 关联帖文ID（必填）
}
```
- **响应参数**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "sessionId": "long",
    "targetUser": {
      "id": "long",
      "nickname": "string",
      "avatarUrl": "string",
      "creditScore": "integer",
      "online": "boolean"
    },
    "itemCard": {
      "id": "long",
      "type": "integer",
      "content": "string",
      "price": "decimal",
      "imageUrl": "string"
    }
  }
}
```

#### 3.1.2 业务逻辑设计

##### 初始化流程
1. **参数校验**：
   - `targetUserId` 不能为当前登录用户（不许给自己发消息）
   - `itemId` 对应的帖文必须存在且状态为"展示中"（`item_status = 0`）
2. **风控拦截**：
   - 检查当前用户信用分 ≥ 60，否则拒绝发起私信（信用风控）
3. **会话创建/查询**：
   - 根据 `(user_id, target_user_id, item_id)` 查询已有会话
   - 若已存在则复用，避免为同一帖文重复创建会话
   - 若不存在则创建新会话
4. **组装意向卡片**：
   - 从 `item` 表读取帖文微缩信息：ID、类型、内容摘要（前20字）、价格、图片缩略图
5. **组装目标用户信息**：
   - 从 `user` 表读取对方昵称、头像、信用分
   - 查询 WebSocket 在线状态（连接池中是否在线）
6. **返回完整会话数据**

##### 会话模型（无独立 session 表设计）
本系统不设独立的 `session` 表，会话由 `message` 表中的 `(sender_id, receiver_id, item_id)` 三元组隐式定义。创建会话时不写入任何数据，第一条消息发送时才首次写入 `message` 表。

**原因**：简化数据模型，会话生命周期由消息的存在与否自然决定。若某会话所有消息被删除，会话自动消失。

### 3.2 快捷短语模块 (`/api/chat/quick-phrases`)

**业务目标**：返回系统预置的交易高频话术列表，用户点击一键发送，降低打字成本。

#### 3.2.1 API 设计

##### 获取快捷话术池
- **接口**: `GET /api/chat/quick-phrases`
- **请求头**: `Authorization: Bearer {token}`
- **响应参数**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "phrases": [
      "几成新？",
      "什么时候方便面交？",
      "在哪里交接？",
      "可以小刀吗？",
      "还能便宜一点吗？",
      "有实拍图吗？",
      "还在吗？",
      "我要了，怎么交易？"
    ]
  }
}
```

#### 3.2.2 业务逻辑设计

##### 话术存储策略
- 快捷短语由系统后台配置文件维护，不建独立表
- 配置在 `application.yaml` 中的 `ctp.chat.quick-phrases` 列表
- 支持不重启更新（Redis 缓存 + 配置中心，可选）

##### 配置示例
```yaml

    quick-phrases:
      - "几成新？"
```

### 3.3 会话列表模块 (`/api/chat/sessions`)

**业务目标**：返回当前用户的所有会话列表，按最后消息时间倒序排列。

#### 3.3.1 API 设计

##### 获取会话列表
- **接口**: `GET /api/chat/sessions`
- **请求头**: `Authorization: Bearer {token}`
- **请求参数**:
```
page: integer  // 页码（默认1）
size: integer  // 每页条数（默认20）
```
- **响应参数**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "list": [
      {
        "sessionId": "string",          // 会话标识（itemId_targetUserId）
        "targetUser": {
          "id": "long",
          "nickname": "string",
          "avatarUrl": "string",
          "online": "boolean"
        },
        "itemId": "long",
        "itemContent": "string",        // 帖文摘要
        "lastMessage": "string",        // 最后一条消息内容
        "lastMessageTime": "datetime",  // 最后消息时间
        "unreadCount": "integer"        // 未读消息数
      }
    ],
    "total": "long"
  }
}
```

#### 3.3.2 业务逻辑设计

##### 会话聚合查询
由于不设独立 session 表，通过 `message` 表聚合查询：
```sql
SELECT
    CASE WHEN sender_id = #{userId} THEN receiver_id ELSE sender_id END AS target_user_id,
    item_id,
    MAX(created_at) AS last_message_time
FROM message
WHERE sender_id = #{userId} OR receiver_id = #{userId}
GROUP BY target_user_id, item_id
ORDER BY last_message_time DESC
```

然后根据 `(target_user_id, item_id)` 补充对方用户信息和帖文摘要。

## 4. 数据模型设计

### 4.1 站内信表 (message)

已建表，核心字段回顾：

| 字段名 | 数据类型 | 说明 |
|:-------|:---------|:-----|
| `id` | BIGINT | 消息ID |
| `sender_id` | BIGINT | 发送方ID |
| `receiver_id` | BIGINT | 接收方ID |
| `item_id` | BIGINT | 关联帖文ID（会话意向带入） |
| `content` | VARCHAR(500) | 消息内容 |
| `is_read` | TINYINT | 0=未读, 1=已读 |
| `created_at` | DATETIME | 发送时间 |

### 4.2 会话隐式模型

会话由 `(sender_id, receiver_id, item_id)` 三元组唯一确定。两个用户围绕同一帖文产生的所有消息属于同一个会话。

**会话唯一标识**：`"{smallerUserId}_{largerUserId}_{itemId}"`（保证同一对用户的同一帖文只有唯一会话）

### 4.3 Redis 数据结构设计

| Key 格式 | 类型 | 用途 | 过期时间 |
|:---------|:-----|:-----|:---------|
| `ws:online:{userId}` | String | 用户在线状态 | 断线后 30s 过期 |

## 5. 安全设计

### 5.1 认证授权
- 所有接口均需 JWT 认证
- 会话初始化时校验操作者身份，防止伪造发送方

### 5.2 业务规则
- 不允许给自己发消息
- 已失效/已成交的帖文不可发起新会话（可选，初期可放宽）
- 信用分 < 60 的用户禁止主动发起私信

### 5.3 隐私保护
- 返回对方信息时仅含昵称、头像、信用分，不含邮箱
- 会话列表不暴露消息的完整历史，仅返回最后一条摘要

## 7. 性能优化设计

### 7.1 数据库优化
- 会话聚合查询利用 `idx_conversation (sender_id, receiver_id)` 索引
- 快捷短语内存缓存，不查数据库

### 7.2 缓存策略
- 快捷短语配置缓存在 Redis，TTL = 24 小时
- 用户在线状态缓存在 Redis，轻量查询

|:-------|:-----|:-----|

| P1 | GET /api/chat/sessions | 会话列表页面依赖 |
