# 离线补偿与未读红点管理 - 后端技术设计文档

## 1. 概述

本文档详细描述校园交易平台离线消息补偿和未读红点管理的后端技术实现方案，包括离线消息增量同步、已读回执、全局未读总数查询等功能模块的技术架构、接口设计、数据模型和安全策略。

**核心设计原则**：在线靠 WS 实时推送，离线靠 HTTP 补偿拉取。通过"已读回执"机制闭环未读与已读状态，确保用户不会漏掉任何消息。

## 2. 技术栈

- **框架**: Spring Boot 4.x
- **安全**: Spring Security + JWT
- **数据库**: MySQL 8.0+
- **缓存**: Redis

## 3. 核心功能模块设计

### 3.1 离线消息增量同步模块 (`/api/chat/messages/sync`)

**业务目标**：用户登录、应用唤醒或 WS 重新连接时，拉取自上次同步点之后的所有新消息，实现离线消息补偿。

#### 3.1.1 API 设计

##### 离线消息拉取
- **接口**: `GET /api/chat/messages/sync`
- **请求头**: `Authorization: Bearer {token}`
- **请求参数**:
```
lastMessageId: long    // 本地最后一条消息ID（可选，首次同步不传）
timestamp: datetime    // 备选：按时间戳同步（与 lastMessageId 二选一）
limit: integer         // 单次拉取上限（默认 100，最大 200）
```
- **响应参数**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "messages": [
      {
        "id": "long",
        "senderId": "long",
        "senderNickname": "string",
        "receiverId": "long",
        "itemId": "long",
        "content": "string",
        "isRead": "integer",
        "createdAt": "datetime"
      }
    ],
    "hasMore": "boolean",     // 是否还有更多未拉取的消息
    "latestMessageId": "long" // 本次返回的最新消息ID（作为下次同步的 lastMessageId）
  }
}
```

#### 3.1.2 业务逻辑设计

##### 同步流程
1. **确定同步起点**：
   - 若传了 `lastMessageId`：查询 `WHERE id > #{lastMessageId}`
   - 否则查询该用户作为发送方或接收方的所有历史消息
2. **查询范围**：
   ```sql
   SELECT * FROM message
   WHERE (sender_id = #{userId} OR receiver_id = #{userId})
     AND id > #{lastMessageId}
   ORDER BY id ASC
   LIMIT #{limit}
   ```
3. **批量限制**：单次最多返回 200 条，超过则 `hasMore = true`，前端需再次请求
4. **返回增量数据** + 最新消息 ID（前端的下次同步锚点）

##### 增量同步 vs 全量同步
- **增量同步**（推荐）：前端维护 `lastMessageId`，只拉取新消息，节省带宽
- **全量同步**：不传 lastMessageId，拉取全部历史（首次安装或清缓存时使用）

##### 同步时机

| 触发场景 | 调用方 |
|:---------|:-------|
| 用户登录成功 | 前端自动静默调用 |
| WebSocket 重连成功 | 前端自动静默调用 |
| 应用从后台唤醒 | 前端自动静默调用 |
| 用户手动下拉刷新 | 聊天页面下拉触发 |

### 3.2 消息已读回执模块 (`/api/chat/session/{sessionId}/read`)

**业务目标**：用户进入会话后，标记该会话中所有发给自己的消息为已读。

#### 3.2.1 API 设计

##### 消息已读回执
- **接口**: `PUT /api/chat/session/{sessionId}/read`
- **请求头**: `Authorization: Bearer {token}`
- **路径参数**:
```
sessionId: string  // 会话标识（格式："{smallerUserId}_{largerUserId}_{itemId}"）
```
- **响应参数**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "markedCount": "integer"  // 已标记为已读的消息数量（用于前端红点清零校验）
  }
}
```

#### 3.2.2 业务逻辑设计

##### 已读回执流程
1. **解析 sessionId**：从 sessionId 中提取 `{userId1}_{userId2}_{itemId}`
2. **标记已读**：
   ```sql
   UPDATE message
   SET is_read = 1
   WHERE item_id = #{itemId}
     AND receiver_id = #{currentUserId}
     AND is_read = 0
   ```
3. **返回影响行数**：供前端校验红点是否清零
4. **通过 WebSocket 通知对方**（可选）：
   - 向发送方推送 `{ "type": "message_read", "data": { "sessionId": "..." } }`
   - 对方可在聊天界面看到"已读"标记

##### 会话标识格式
```
sessionId = min(userId1, userId2) + "_" + max(userId1, userId2) + "_" + itemId
```
例如用户 3 和用户 7 围绕帖文 15 的会话：`"3_7_15"`

### 3.3 全局未读总数模块 (`/api/chat/unread-count`)

**业务目标**：汇总查询当前用户所有未读消息数量，用于首页 Tab 红点气泡和会话列表未读数。

#### 3.3.1 API 设计

##### 获取全局未读总数
- **接口**: `GET /api/chat/unread-count`
- **请求头**: `Authorization: Bearer {token}`
- **响应参数**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "totalUnread": "integer",
    "sessions": [
      {
        "sessionId": "string",
        "targetUserId": "long",
        "targetNickname": "string",
        "itemId": "long",
        "unreadCount": "integer",
        "lastMessagePreview": "string"
      }
    ]
  }
}
```

#### 3.3.2 业务逻辑设计

##### 未读统计查询
```sql
-- 全局未读总数
SELECT COUNT(1)
FROM message
WHERE receiver_id = #{userId}
  AND is_read = 0

-- 按会话分组未读数
SELECT
    sender_id,
    item_id,
    COUNT(1) AS unread_count,
    MAX(created_at) AS latest_time,
    (SELECT content FROM message m2
     WHERE m2.sender_id = m.sender_id
       AND m2.item_id = m.item_id
     ORDER BY m2.created_at DESC LIMIT 1) AS last_message
FROM message m
WHERE receiver_id = #{userId}
  AND is_read = 0
GROUP BY sender_id, item_id
ORDER BY latest_time DESC
```

##### 性能优化
- 利用索引 `idx_receiver_read (receiver_id, is_read)` 加速查询
- 全局未读总数可缓存 Redis，TTL 30 秒（用户收到新消息时主动失效）

## 4. 数据模型设计

### 4.1 站内信表 (message)

未读管理核心字段：

| 字段名 | 数据类型 | 默认值 | 说明 |
|:-------|:---------|:-------|:-----|
| `id` | BIGINT | AUTO_INC | 消息ID（增量同步锚点） |
| `sender_id` | BIGINT | — | 发送方ID |
| `receiver_id` | BIGINT | — | 接收方ID（已读回执条件） |
| `item_id` | BIGINT | — | 关联帖文ID（会话分组键） |
| `is_read` | TINYINT | 0 | 0=未读, 1=已读 |
| `created_at` | DATETIME | NOW() | 发送时间 |

### 4.2 核心索引

| 索引 | 用途 | 查询场景 |
|:-----|:-----|:---------|
| `idx_receiver_read (receiver_id, is_read)` | 未读统计 | 全局未读总数 + 按会话分组 |
| `idx_conversation (sender_id, receiver_id)` | 会话聚合 | 历史消息查询 |
| 主键 `id` | 增量同步 | `WHERE id > #{lastMessageId}` |

### 4.3 Redis 数据结构设计

| Key 格式 | 类型 | 用途 | 过期时间 |
|:---------|:-----|:-----|:---------|
| `unread:count:{userId}` | String | 全局未读总数缓存 | 30s |

## 5. 安全设计

### 5.1 认证授权
- 所有接口均需 JWT 认证
- 已读回执只能标记发给自己的消息（`receiver_id = 当前用户`）
- 离线同步只能拉取自己的消息（`sender_id = 自己 OR receiver_id = 自己`）

### 5.2 数据权限
- 不暴露他人之间的消息
- 会话列表不返回对方的未读数

## 7. 前端本地同步策略（参考）

### 7.1 消息本地持久化

```
在线路径:  WS 收到消息 → 追加到 Vuex/本地数组 → 写入 IndexedDB
离线路径:  登录/唤醒 → 调 sync 接口 → 合并去重 → 写入 IndexedDB
```

### 7.2 红点状态机

```
收到新消息 → is_read=0 → 未读计数+1 → 更新会话列表红点 + 底部Tab红点
进入会话页 → 调 read 接口 → 后端标记已读 → 前端清零会话红点 + 重算Tab红点
离开会话页 → 不处理
```

### 7.3 去重策略
- 前端维护已接收的 messageId Set
- sync 回来的消息按 ID 去重再合并
- WS 实时消息与 sync 消息可能存在重叠（网络切换时），按 ID 去重

## 8. 性能优化设计

### 8.1 数据库优化
- 利用索引 `idx_receiver_read` 加速未读查询
- 增量同步走主键索引 `id`，性能极高
- 已读回执 UPDATE 带上 `is_read = 0` 条件，避免重复更新

### 8.2 缓存策略
- 全局未读总数缓存 Redis，TTL 30 秒
- 进入会话后，该会话的未读数直接改为 0（无需查库确认）
- 收到新消息时，主动失效未读缓存
