# 生命周期自动化管理 - 后端技术设计文档

## 1. 概述

本文档详细描述校园交易平台帖文生命周期自动化管理的后端技术实现方案。通过定时任务自动下架过期帖文，避免"僵尸信息"长期占据首页视图，保持平台信息的时效性和真实性。

**核心设计原则**：自动化、低侵入、可观测。系统静默执行，用户无需感知，仅在"我的发布"列表可看到帖文状态变更为"已失效"。

## 2. 技术栈

- **框架**: Spring Boot 4.x
- **调度**: Spring Scheduled（`@Scheduled`）
- **数据库**: MySQL 8.0+

## 3. 核心功能模块设计

### 3.1 过期帖文自动下架任务

**业务目标**：定时扫描 item 表中已过期但仍处于"展示中"状态的帖文，批量更新为"已失效"。

#### 3.1.1 定时任务设计

##### 任务配置
```java
@Scheduled(cron = "0 0 3 * * ?") // 每天凌晨 3:00 执行
public void expireOverdueItems() {
    // 分批更新逻辑
}
```

##### Cron 表达式说明
```
0 0 3 * * ?
│ │ │ │ │ │
│ │ │ │ │ └─ 星期（任意）
│ │ │ │ └─── 月份（任意）
│ │ │ └───── 日期（任意）
│ │ └─────── 小时（凌晨3点）
│ └───────── 分钟（0分）
└─────────── 秒（0秒）
```

**选择凌晨 3:00 的原因**：低峰期，用户活跃度最低，减少对正常业务的影响。

#### 3.1.2 业务逻辑设计

##### 执行流程
1. 查询待失效帖文总数：
```sql
SELECT COUNT(1) FROM item WHERE item_status = 0 AND expire_at <= NOW()
```
2. 若总数 = 0，记录日志并结束
3. 分批更新（每批 500 条）：
```sql
UPDATE item
SET item_status = 2
WHERE id IN (
    SELECT id FROM item
    WHERE item_status = 0 AND expire_at <= NOW()
    LIMIT 500
)
```
4. 循环执行直到无过期记录
5. 记录执行日志（处理总数、耗时、成功/失败）

##### 分批更新原因
- 避免单次 UPDATE 锁住大量行，影响正常业务读写
- 每批 500 条，批次之间间隔 100ms，给其他事务执行机会
- 若数据量持续增长，可调整为每批 200 条

##### 伪代码
```java
@Scheduled(cron = "0 0 3 * * ?")
public void expireOverdueItems() {
    log.info("开始执行过期帖文自动下架任务");
    long startTime = System.currentTimeMillis();
    int totalExpired = 0;

    while (true) {
        int affected = itemMapper.expireBatch(500);
        if (affected == 0) break;
        totalExpired += affected;
        // 批次间隔，释放数据库锁
        Thread.sleep(100);
    }

    long elapsed = System.currentTimeMillis() - startTime;
    log.info("过期帖文自动下架完成，共处理 {} 条，耗时 {} ms", totalExpired, elapsed);
}
```

### 3.2 Mapper SQL 设计

#### 3.2.1 分批失效 SQL
```java
@Update("""
    UPDATE item
    SET item_status = 2
    WHERE id IN (
        SELECT id FROM (
            SELECT id FROM item
            WHERE item_status = 0 AND expire_at <= NOW()
            LIMIT #{batchSize}
        ) AS tmp
    )
    """)
int expireBatch(@Param("batchSize") int batchSize);
```

**注意**：MySQL 不允许在 UPDATE 子查询中直接 LIMIT 同一张表，需要使用嵌套子查询（`SELECT ... FROM (SELECT ...) AS tmp`）绕过限制。

### 3.3 配置化设计

定时任务相关配置项：

```yaml
ctp:
  item:
    expire:
      cron: "0 0 3 * * ?"   # Cron 表达式
      batch-size: 500        # 每批处理数量
      batch-interval-ms: 100 # 批次间隔（毫秒）
```

可通过 `@ConfigurationProperties` 绑定，支持不重启调整参数。

## 4. 数据模型设计

### 4.1 涉及的字段（item 表）

| 字段名 | 数据类型 | 说明 |
|:-------|:---------|:-----|
| `item_status` | TINYINT | 0=展示中, 2=已失效 |
| `expire_at` | DATETIME | 过期时间（发布时 +14 天） |

### 4.2 状态变更

| 触发方式 | 变更前 | 变更后 | 说明 |
|:---------|:-------|:-------|:-----|
| 定时任务（自动） | item_status = 0 | item_status = 2 | expire_at <= NOW() |
| 用户操作（手动） | item_status = 0 | item_status = 2 | 用户点击"提前下架" |

两条路径殊途同归，最终状态一致。

---

## 5. 安全设计

### 5.1 并发安全
- 使用 MySQL 行锁，UPDATE 时自动加锁，防止重复更新
- 分批 + 间隔执行，避免长事务锁表

### 5.2 幂等性
- 条件 `item_status = 0` 确保已失效的帖文不会被重复更新
- 任务可安全重入，多次执行不会产生副作用

### 5.3 异常处理
- 任务异常不中断后续批次
- 异常信息记录日志，可接入监控告警

## 7. 性能优化设计

### 7.1 数据库优化
- 利用 `idx_search` 联合索引的 `item_status` 前缀加速查询
- 建议额外建立 `idx_expire (expire_at)` 索引，加速过期帖文扫描
- 分批更新，每批 500 条，批次间隔 100ms

### 7.2 资源消耗
- 假设日活跃帖文 5000 条，日均过期约 350 条（1批即可完成）
- 执行耗时预计 < 100ms，对数据库影响极小
- 即使帖文量增长到 10 万条，分 200 批完成，总耗时约 20 秒

---

## 8. 前端影响

- **无直接用户界面**，静默执行
- 用户仅在"我的发布"列表中看到帖子状态变为"已失效"（非高亮灰色显示）
- 已失效的帖文从首页信息流自动移除
