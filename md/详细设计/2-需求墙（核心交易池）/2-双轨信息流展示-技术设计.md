# 双轨信息流展示（首页信息池）- 后端技术设计文档

## 1. 概述

本文档详细描述校园交易平台首页信息流（双轨信息池）的后端技术实现方案。首页通过"求购"/"转让"两个独立 Tab 展示信息卡片列表，支持无限滚动分页，确保首页信息纯净度。

**核心设计原则**：数据防腐——严格过滤非法状态帖文，确保首页只展示有效可交易的信息。

## 2. 技术栈

- **框架**: Spring Boot 4.x
- **安全**: Spring Security + JWT
- **数据库**: MySQL 8.0+
- **缓存**: Redis

## 3. 核心功能模块设计

### 3.1 需求墙信息流模块 (`/api/item/feed`)

#### 3.1.1 API 设计

##### 获取需求墙列表
- **接口**: `GET /api/item/feed`
- **请求头**: `Authorization: Bearer {token}`
- **请求参数**:
```
type: integer   // 1=求购, 2=转让（必填）
page: integer   // 页码（从1开始，默认1）
size: integer   // 每页条数（默认20，最大50）
```
- **响应参数**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "list": [
      {
        "id": "long",
        "type": "integer",              // 1=求购, 2=转让
        "categoryName": "string",       // 分类名称
        "campus": "string",             // 校区/楼栋
        "price": "decimal",             // 期望价格（null=面议）
        "content": "string",            // 文本描述
        "imageUrl": "string",           // 缩略图URL
        "publisherId": "long",          // 发布者ID
        "publisherNickname": "string",  // 发布者昵称
        "publisherCreditScore": "integer", // 发布者信用分
        "createdAt": "datetime"         // 发布时间
      }
    ],
    "total": "long",
    "page": "integer",
    "size": "integer"
  }
}
```

#### 3.1.2 业务逻辑设计

##### 信息流查询流程
1. **参数校验**：`type` 必须为 1 或 2，page/size 使用默认值
2. **数据过滤（防腐层）**：
   - `item_status = 0`（仅展示中）
   - `expire_at > NOW()`（未过期）
   - `type = {入参}`（求购或转让）
3. **排序规则**：按 `created_at` 降序（最新在前，擦亮过的帖文 `created_at` 被更新，自然出现在顶部）
4. **分页查询**：MyBatis 分页查询，返回 `list + total`
5. **关联查询**：
   - 联表 `user` → 获取发布者昵称、信用分
   - 联表 `category` → 获取分类名称
6. **防腐处理**：后端强制过滤非法状态数据，即使前端传参绕过，后端仍保证首页纯净

##### SQL 伪代码
```sql
SELECT i.*, u.nickname, u.credit_score, c.name AS category_name
FROM item i
JOIN user u ON i.user_id = u.id
JOIN category c ON i.category_id = c.id
WHERE i.item_status = 0
  AND i.expire_at > NOW()
  AND i.type = #{type}
ORDER BY i.created_at DESC
LIMIT #{offset}, #{size}
```

##### 分页设计
- **滚动分页**：前端采用触底加载更多（Infinite Scroll），不使用传统页码点击
- **防重复**：前端维护已加载 ID 集合，服务端保证分页连续性
- **每页默认 20 条**，最大 50 条，防止单次拉取过多数据

### 3.2 信息卡片数据组装

#### 3.2.1 卡片字段说明

| 字段 | 来源 | 说明 |
|:-----|:-----|:-----|
| `id` | item.id | 帖文唯一标识 |
| `type` | item.type | 1=求购, 2=转让（前端据此渲染不同颜色标签） |
| `categoryName` | category.name | 物品分类名称 |
| `campus` | item.campus | 交接校区/楼栋 |
| `price` | item.price | 期望价格，null 时前端渲染"面议" |
| `content` | item.content | 文本描述，前端超过 3 行折叠 |
| `imageUrl` | item.image_url | 缩略图 URL，有值则显示，无值则隐藏图片区 |
| `publisherNickname` | user.nickname | 发布者昵称 |
| `publisherCreditScore` | user.credit_score | 信用分，前端据此渲染不同颜色徽章 |
| `createdAt` | item.created_at | 发布时间，用于排序和前端展示"x小时前" |

#### 3.2.2 数据量控制
- 单次查询最大返回 50 条
- 超过 50 条时提示"仅展示最近50条"
- total 字段用于前端判断是否还有更多数据

## 4. 数据模型设计

### 4.1 涉及的数据库表

| 表 | 角色 | 关键字段 |
|:---|:-----|:---------|
| `item` | 主表 | id, user_id, type, category_id, campus, price, content, image_url, item_status, created_at, expire_at |
| `user` | 发布者信息 | nickname, credit_score |
| `category` | 分类名称 | name |

### 4.2 核心索引

利用 item 表已有联合索引：
- `idx_search (item_status, type, category_id, campus)` — 信息流查询主导索引
- `idx_user_id (user_id)` — 按发布者查询

---

## 5. 安全设计

### 5.1 认证授权
- 信息流接口需 JWT 认证（确保只有校内用户可浏览）
- 不暴露 `password_hash` 等敏感字段

### 5.2 数据防腐
- 后端强制过滤已成交（status=1）和已失效（status=2）的帖文
- 后端强制过滤过期帖文（expire_at <= NOW()）
- 不信任前端传入的任何状态参数

---

## 6. 异常处理设计

### 6.1 统一异常处理
复用现有 `GlobalExceptionHandler`：

- `BusinessException`: 业务异常（如 type 参数非法）
- `ValidationException`: 参数校验异常（page/size 超出范围）

### 6.2 错误码设计

```java
FEED_TYPE_INVALID(40001, "信息流类型参数非法，必须为1(求购)或2(转让)"),
PAGE_SIZE_EXCEEDED(40002, "单次查询条数不能超过50"),
```

---

## 7. 性能优化设计

### 7.1 数据库优化
- 利用联合索引 `idx_search` 加速查询
- 分页使用 `LIMIT offset, size`，避免全表扫描
- 联表查询控制在 3 张表以内

### 7.2 缓存策略（可选）
- 首页高频访问，可对第 1 页数据做短时 Redis 缓存（如 30 秒）
- 有用户发布新帖时主动失效缓存

### 7.3 图片优化
- 列表中的图片展示缩略图（已压缩至 WebP）
- 前端懒加载图片，减少首屏带宽消耗
