# `campus-trading-platform` 产品需求分析报告

## 一、 产品背景与概述
*   **项目名称**：campus-trading-platform (校园跳蚤市场)
*   **产品定位**：一个专注校园内部、重信息撮合、轻图片展示的二手交易平台。
*   **核心痛点**：传统二手平台发布门槛高（需精美修图）、无关信息多、陌生人交易信任机制薄弱。
*   **核心价值**：通过“纯文本流+结构化标签+同校实名+严格信用计算”，实现极低门槛发布与快速、安全的线下撮合。

## 二、 核心业务流程（撮合闭环）
1.  **身份认证**：用户通过校园专属凭证（教育邮箱）注册实名账号。
2.  **信息发布**：用户在需求墙发布带有多级标签的短文本“求购”或“转让”信息。
3.  **信息检索**：目标群体通过关键词或多维分类筛选获取到该信息。
4.  **意向沟通**：浏览者查看发布者信用分，无异议后通过WebSocket站内信发起实时沟通。
5.  **线下履约与评价**：双方约定线下交收。完成后，系统触发双向评价，底层重新核算用户信用分，形成信任闭环。

## 三、 功能需求详细拆解

### 1. 账号与个人中心
*   **校园实名认证**：必须通过.edu.cn邮箱进行注册验证，确保100%校内用户。
*   **个人主页**：展示用户基础信息（自动屏蔽真实姓名）、当前信用分徽章及历史评价记录。
*   **状态管理**：管理自己发布的帖文（擦亮置顶、提前下架、标记为已成交）。

### 2. 需求墙（核心交易池）
*   **双轨信息流**：首页核心分为“我要求购”与“我要转让”两个独立Tab。
*   **克制化发布**：
    *   **文本限制**：最多200字纯文本，强制屏蔽违规词。
    *   **图片限制**：最多允许上传1张实物图作为补充（后台需进行强压缩转缩略图）。
*   **结构化打标**：发布时强制选择[物品分类]、[期望价格区间]、[交接校区/楼栋]。
*   **生命周期管理**：信息默认上架14天，逾期未成交则自动转为“已失效”状态并隐藏。

### 3. 分类索引（筛选引擎）
*   **模糊关键词搜索**：支持对所有有效帖文的“文本描述”进行全文检索。
*   **多维复合筛选**：支持 [交易类型] + [分类] + [校区] + [价格区间] 的并集查询，快速定位目标。

### 4. WebSocket站内信（实时沟通）
*   **意向带入**：从需求墙点击“聊一聊”进入会话时，顶部自动悬浮“当前 обсуждаю的物品卡片”，降低沟通成本。
*   **实时收发**：基于WebSocket实现文字消息的实时投递、未读红点提示及在线状态感知。
*   **离线补偿**：断线期间的消息由系统暂存至队列，用户再次登录时通过接口全量拉取离线消息并持久化到本地。
*   **快捷短语**：平台内置交易高频话术库（如“几成新？”、“何时面交？”）供一键发送。

### 5. 评价与信用分系统
*   **交易打闭环**：发起方标记“已成交”并关联对话用户后，系统下发站内信通知双方互评。
*   **多维评价体系**：采用“标签点选（如：爽快、描述不符）+ 简短文字说明”的形式。
*   **动态积分引擎**：
    *   新用户获基础分（如100分）。
    *   正向行为（成交+1、好评+2）加分；负面行为（放鸽子被核实-5，差评-2）扣分。
*   **信用风控拦截**：信用分直观外显为徽章。信用分跌破阈值（如60分）的账号，将被系统自动剥夺“发布”与“主动私信”的权限。

## 四、 非功能性需求建议
*   **架构选型**：由于不强调高并发与大图片传输，初期采用单体架构（如Spring Boot / Node.js + MySQL）即可快速试错。
*   **模块解耦**：站内信模块（WebSocket）未来最容易产生性能瓶颈，代码组织上需与主业务逻辑（发布、浏览）解耦，为后续独立拆分微服务做准备。
*   **性能考量**：若后期文字检索量猛增，“分类索引”模块的模糊查询可考虑从MySQL的 `LIKE` 迁移至 Elasticsearch 平滑支撑。
